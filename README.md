# My-eBookStore 网上书店系统

My-eBookStore 是面向软件工程课程设计的网上书店系统，围绕“普通用户购书、书店经营管理、平台统一监管”三类核心场景构建。项目参考需求报告与需求规格说明书实现，覆盖图书浏览与搜索、个性化推荐、购物车、订单支付、促销活动、会员积分、用户评价、后台管理、平台统计分析等完整业务链路。

系统采用前后端分离的工程结构，前端提供用户端与后台管理端页面，后端提供统一 `/api` 接口，数据库使用 SQL Server 保存用户、店铺、图书、订单、促销、评价、积分、推荐配置等业务数据。项目目标不是单页演示，而是形成一个可运行、可联调、可扩展的课程设计级业务系统。

## 项目定位

本项目面向一个多角色、多店铺的线上图书交易平台：

- 普通用户可以注册登录、浏览图书、搜索图书、查看详情、加入购物车、提交订单、模拟支付、查看订单、参与促销、管理地址、积累积分并发表评价。
- 书店管理员可以维护本店图书、查看本店订单、管理店铺信息、参与平台活动、查看本店经营统计。
- 平台管理员可以管理全平台用户、店铺、订单、图书、促销、推荐策略，并查看平台统计和风险分析。

系统强调业务闭环：从用户搜索、推荐、下单、支付，到库存扣减、销量累加、积分发放、评价生成、后台统计，数据均进入数据库并通过接口驱动前端展示。

## 核心功能

### 用户端

- 用户注册、登录、退出与 JWT 登录态维护。
- 图书分类浏览、书名搜索、作者搜索、ISBN 精确搜索。
- 图书详情展示，包括作者、出版社、ISBN、价格、库存、店铺、评分、评价和相关图书。
- 首页推荐，包括“猜你喜欢”和“热销图书”。
- 购物车增删改查、结算页确认、地址选择、优惠券使用。
- 订单创建、模拟支付、取消订单、申请退款、订单详情查看。
- 购买后评价，限制未购买或重复评价。
- 促销活动参与、签到、周代金券领取、积分流水查看、积分奖品兑换。
- 个人中心维护昵称、联系方式、地址、积分和订单信息。

### 书店管理员端

- 登录后台并进入书店管理视图。
- 新增、编辑、上下架本店图书，维护价格、库存、ISBN、分类和描述信息。
- 查看本店订单和订单详情，处理订单状态与退款流程。
- 维护店铺名称、简介等店铺资料。
- 查看本店用户与经营统计数据。
- 参与平台促销活动，配置参与书目和店铺券。

### 平台管理员端

- 管理全平台用户状态，支持封禁、恢复等操作。
- 管理店铺状态，支持正常、封禁等平台级控制。
- 查看全平台订单、订单状态和退款处理。
- 强制下架违规图书。
- 创建和维护促销活动、平台代金券、积分兑换奖品。
- 查看平台数据统计、风险店铺分析和报表导出。
- 管理推荐策略配置，包括首页推荐、热销权重、搜索推荐和详情相关推荐开关。

## 推荐与搜索设计

项目将推荐能力拆分为三类业务场景：

- 首页“猜你喜欢”：根据当前用户最近 5 条搜索历史的关键词 embedding 计算相似度，按距离总和最小筛选图书；同一本书在多个店铺出现时，保留销量最高的店铺商品。
- 首页“热销图书”：按在售图书销量排序，展示销量最高的图书商品。
- 搜索推荐：书名搜索使用关键词 embedding 对候选图书排序；作者搜索使用作者字段子串匹配；ISBN 搜索使用完全相等匹配。
- 详情页相关图书：选取当前图书同分类、同店铺、在售且销量较高的图书作为相关推荐。

embedding 模型通过独立接口封装，真实模型配置文件不提交仓库，避免泄露模型 URL、API Key 等敏感信息。没有真实模型时，系统保留本地确定性 fallback，保证开发、联调和测试阶段可以稳定运行。

## 数据库设计

数据库使用 SQL Server，围绕需求规格中的实体进行建模，主要包括：

- 用户体系：普通用户、书店管理员、平台管理员、用户状态、会员等级、积分。
- 店铺体系：店铺信息、店铺状态、店铺与管理员关系。
- 图书体系：图书分类、图书基础信息、ISBN、embedding、店铺图书商品、库存、销量、上下架状态。
- 交易体系：购物车、收货地址、订单、订单明细、支付记录、退款记录。
- 促销体系：促销活动、优惠券、用户优惠券、店铺活动参与、活动书目、积分奖品。
- 互动体系：搜索历史、用户评价、积分流水。
- 平台配置：推荐策略配置、统计分析相关数据。

数据库脚本集中在 `database/` 目录，包含建表、索引、种子数据、存储过程、触发器和测试数据脚本。后端运行时代码通过 DAO 层访问数据库，业务路由不直接编写 SQL，降低后续维护成本。

## 技术架构

项目结构如下：

```text
My-eBookStore/
├── backend/          # FastAPI 后端应用
│   ├── app/          # API 路由、配置、认证、数据库连接、DAO、推荐逻辑
│   ├── config/       # embedding 等敏感配置 sample
│   └── requirements.txt
├── database/         # SQL Server 数据库脚本
├── frontend/         # 前台用户端与后台管理端页面
│   ├── admin/        # 后台管理页面
│   ├── css/          # 页面样式
│   └── js/           # 前端 API 封装与页面逻辑
├── scripts/          # 数据库检查、初始化、SQL 执行、冒烟联调脚本
├── tests/            # 自动化测试
└── README.md
```

后端使用 FastAPI 组织接口，使用 `pyodbc` 连接 SQL Server，通过 DAO 层封装数据库访问。前端使用原生 HTML、CSS、JavaScript 构建页面，通过统一的 `api.js` 调用后端接口。接口响应统一为：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

这种结构使前端、后端、数据库、测试职责边界清晰，便于多人分工。

## 权限与角色控制

系统按需求规格划分三类角色：

- `customer`：普通用户，对应前台购书、订单、促销和个人中心能力。
- `seller`：书店管理员，对应本店图书、订单、店铺、活动参与和统计能力。
- `platform_admin`：平台管理员，对应全平台用户、店铺、订单、促销、推荐和统计能力。

后端通过 JWT 识别当前用户，通过角色依赖控制接口访问权限。卖家接口会限制店铺边界，避免越权操作其他店铺数据；平台管理员接口用于跨店铺管理和平台级监管。

## 交易一致性

订单和支付相关操作采用后端事务处理。支付成功后，系统会在同一业务流程中完成：

- 更新订单状态和支付状态。
- 写入支付记录。
- 扣减图书库存。
- 增加图书销量。
- 清理购物车中已下单商品。
- 按实付金额发放积分并写入积分流水。

这避免了前端模拟支付导致的数据不一致问题，也保证后台订单、库存、销量和用户积分可以同步刷新。

## 促销与会员体系

促销模块覆盖需求文档中的签到、代金券、会员等级、积分兑换、活动参与等场景：

- 用户每日签到获得积分。
- 高等级用户可领取周代金券。
- 用户可参与平台活动，获得积分或优惠券奖励。
- 用户可使用积分兑换奖品。
- 管理员可维护平台代金券、活动类型和积分奖品。
- 书店管理员可选择是否参与活动，并配置活动书目。

该模块把“购买交易”和“用户活跃”连接起来，有利于体现平台运营能力。

## 后台统计与风险分析

后台统计面向书店管理员和平台管理员提供不同视角：

- 书店管理员查看本店销售趋势、订单量、销售额和热门图书。
- 平台管理员查看全平台用户、店铺、订单、销售、退款等统计数据。
- 风险分析根据订单量、退款率、重复购买和短时间订单集中度等因素计算风险值。
- 报表导出支持将统计数据导出为 CSV，便于课程展示和后续分析。

## 项目优点

### 1. 业务覆盖完整

项目不是单纯的图书展示 demo，而是覆盖了用户、书店、平台三方角色的完整业务流程。用户端、卖家端、平台端之间通过真实数据库联动，能够展示从浏览到交易再到后台管理的闭环。

### 2. 数据库驱动，减少前端假数据依赖

核心列表、订单、购物车、支付、促销、积分、评价、统计数据均通过后端接口从 SQL Server 获取。测试数据独立放在数据库脚本中，便于演示和联调，也便于后续替换为正式数据。

### 3. 推荐算法有明确业务规则

项目将推荐算法与业务场景绑定，而不是简单随机推荐。首页、搜索页、详情页分别使用不同策略，且预留真实 embedding 模型接口，便于后续接入更强的语义检索能力。

### 4. 后端结构便于维护

后端按路由、认证、响应、数据库连接、DAO、embedding 模块拆分。SQL 查询集中在 DAO 层，业务路由主要负责参数、权限和流程编排，降低了后续修改数据库或接口时的影响范围。

### 5. 交易流程注重一致性

支付、库存、销量、积分、购物车清理等关键数据变更由后端事务统一处理，避免前端状态和数据库状态不一致。该设计更接近真实业务系统，而不是只在页面上模拟成功。

### 6. 支持多人协作

项目目录按职责拆分清楚：前端、后端、数据库、脚本、测试分别独立。`scripts/` 和 `tests/` 已放在项目根目录，方便后续由专人负责数据库检查、冒烟测试和自动化测试。

### 7. 隐私配置隔离

真实 embedding 模型的 URL、API Key 等敏感信息不进入仓库，只保留 sample 配置文件。这样既能保留模型接入能力，也能降低密钥泄露风险。

## 当前实现范围

当前项目已覆盖需求规格中的主要业务模块：

- 用户认证与角色权限。
- 图书分类、浏览、搜索、详情和推荐。
- 购物车、地址、订单、支付、取消、退款。
- 评价、积分、优惠券、签到、活动参与、积分兑换。
- 店铺、图书、订单、用户、促销、统计等后台管理能力。
- SQL Server 建表、索引、种子数据、测试数据。
- 自动化测试和冒烟联调脚本。

后续如果需求继续扩展，可以在现有结构上继续补充真实支付、更加复杂的推荐策略、更多报表维度、风控规则和运营活动配置。

## Docker 部署

项目根目录的 `compose.yaml` 会启动 SQL Server、数据库初始化任务和 FastAPI 应用。生产配置保存在不会提交到 Git 的 `backend/.env`，可从 `backend/.env.sample` 创建。

```bash
docker compose up -d --build
docker compose ps
```

应用仅监听宿主机回环地址 `http://127.0.0.1:8003`，适合由宿主机上的反向代理或 Cloudflare Tunnel 转发。停止服务时运行：

```bash
docker compose down
```

SQL Server 数据保存在 Compose 命名卷中；不要使用 `docker compose down -v`，除非明确需要删除全部业务数据并重新初始化。
